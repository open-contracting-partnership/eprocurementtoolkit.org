e-Registration

Registration

No. Requirement Additional description Priority
1Non-authenticated (supplier) users must be able to register their EO details in the system.The system should not allow double-entries.Mandatory
2Non-authenticated (supplier) users could be able to retrieve their EO details from external sources (e.g. National Supplier Registry system).N/ADesirable
3System must validate the organization/user profile details prior to registering them.N/AMandatory
4Administrator users should be able to register EOs on the system.The system must not allow double-entries.Important
5Administrator users must be able to view the details of registered EO and CA profiles.N/AMandatory
6EO users should be notified about their EO registration approval outcome.N/AImportant
7EO users must be able to register other users of their EO.N/AMandatory
8EO users must be able to view/modify the profiles of their organization and respective users.Restrictions may apply (e.g. a user cannot change the password of another user when an EO changes critical organization details such as the organization type, re-approval by Administrators might be required).Mandatory
9Administrator users should be able to register CAs in the system.N/AImportant
10System could allow Administrator users to define the hierarchy of CAs by defining parent-child relationships between CAs.N/ADesirable
11Administrator users should be able to register other Administrator users.N/AImportant
12Administrator users must be able to register CA users.N/AMandatory
13Users could receive a confirmation upon successful profile registration, together with details for activating their account.Their username should be provided in the confirmation message.Desirable
14Administrator users should be able to deactivate organization/user profiles, forbidding such organizations/users to authenticate.Through this mechanism the system can support the blacklisting of EOs, forbidding them from authenticating in the system.Important
15System must validate the uniqueness of the identity of each organization undergoing the registration process.N/AMandatory

Authentication

No. Requirement Additional description Priority
16Users must be able to authenticate themselves in the system (i.e. login).N/AMandatory
17System could allow the authentication of users having their profiles in an external user registry system (e.g. e-ID infrastructure).N/ADesirable
18System could support user authentication through Single Sign-On (SSO) services.N/ADesirable
19System should automatically logout authenticated users if they remain inactive for a pre-specified period of time (i.e. session timeout).N/AImportant
20Non-authenticated users must have access only to information that is publicly available.N/AMandatory
21A "forgot password" mechanism should be made available to the users.N/AImportant
22Users should be locked out of the system after a specific number of failed log-in attempts.N/AImportant
23Re-activating a "locked" account must be possible to Administrator users.N/AMandatory
24Users could be obliged to accept a User Agreement / Terms and Conditions text before obtaining access to the full system functionality.N/ADesirable

Authorization

No. Requirement Additional description Priority
25System must be built following a role-based architecture.N/AMandatory
26Authenticated users must have access to information and functionality according to their respective role.N/AMandatory
27Authenticated users could have a personalized task-list to follow up.N/ADesirable

Management

No. Requirement Additional description Priority
28Administrator users must be able to review and approve/reject EO registrations.Only approved companies may submit bids. The status for a company registration may change at any time.Mandatory
29Users must be able to modify specific information of their profile.Some critical information (e.g. Role) should only be managed by authorized personnel (e.g. Administrator users).Mandatory
30Administrator users must be able to modify the user role of a user.N/AMandatory
31Administrator users must be able to modify specific information in a user profile.N/AMandatory
32Administrator users should be able to de-activate a user account.N/AImportant
33Users must be able to view a CA profile.N/AMandatory
34Administrator users must be able to modify CA profile details.N/AMandatory
35Administrator and CA users must be able to view EO details.N/AMandatory
36Administrator users must be able to modify EO details.N/AMandatory
37Administrator users could be able to define password rules (e.g. acceptable length, expiration period, etc.).N/ADesirable

Searching

Organizations

No. Requirement Additional description Priority
38Administrator Users must be able to search for Organizations.Typical search parameters include: name, address.Mandatory
39CA Users must be able to search for Organizations.Typical search parameters include: name, address, CPV, geographical coverage, national/international vendor, preferred supplier status.Mandatory
40EO Users must be able to search for CAs.EO users should not be allowed to search/view the profile of other EOs. Typical search parameters include: name, address.Mandatory
41Non-authenticated users should be able to search for CAs.Typical search parameters include: name, address.Important

Users

No. Requirement Additional description Priority
42Administrator Users must be able to search for Users.Typical search parameters include: name, username, email, organization.Mandatory
43CA Users must be able to search for Users.There could be controls for CA users to be able to search/view only designated "contact point" users of other organizations rather than the full set of users. Typical search parameters include: name, username, email, organization.Mandatory
44EO Users must be able to search for Users of their EO as well as CA Users.There must be controls for EO users to be able to search/view only designated "contact point" users of CA rather than the full set of users. Typical search parameters include: name, username, email, organization.Mandatory
45It must be impossible for non-authenticated users to search/view user profiles.N/AMandatory

e-Catalogue

Catalogue workspace management

No. Requirement Additional description Priority
46Administrator users must be able to create manually a Catalogue Workspace in the system under which catalogues will be uploaded.Indicative information to be captured comprises:- Title- Start / End Date- Description- Classification (UNSPSC/CPV)- Catalogue Approver users (i.e. users authorized to approve uploaded catalogue content)- File attachment.Mandatory
47System must validate that all 0 information is provided, in order to properly create the Catalogue Workspace.N/AMandatory
48System must produce an error message and reject the creation of a Catalogue Workspace, in case 0 information is missing from the submitted Catalogue Workspace form.N/AMandatory
49Administrator users must be able to assign the Catalogue Workspace Manager (CWM) role to individual CA users for a given Catalogue Workspace.A CWM user is responsible for the management of the Catalogue Workspace (e.g. manage the start/end dates of the workspace/contract).Mandatory
50Administrator users must be able to assign the Catalogue Approver role to individual CA users for a given Catalogue Workspace.A Catalogue Approver user is responsible for validating new/updated eCatalogues with a view to either accept them for purchasing or reject them for further updates by EOs.Mandatory
51Administrator users must be able to assign the Requisitioner role to individual CA users for a given Catalogue Workspace.A Requisitioner user is enabled to create requisitions containing catalogue items of the given Catalogue Workspace.Mandatory
52Administrator users must be able to assign the Purchase Order Manager (POM) role to individual CA users for a given Catalogue Workspace.A POM user is responsible to approve/manage Purchase Orders containing catalogue items of the given Catalogue Workspace.Mandatory
53CA and EO users associated with the Catalogue Workspace must be allowed to view its details.N/AMandatory
54CWM users must be able to modify the non-critical details of a Catalogue Workspace.N/AMandatory
55EO Users must be able to upload a catalogue within a Catalogue Workspace.N/AMandatory
56CWM users could be allowed to upload a catalogue within a Catalogue Workspace on behalf of a EO.N/ADesirable
57System must allow the uploading of an unlimited number of Catalogues.N/AMandatory
58System could impose a limitation on the size (in terms of Megabytes) acceptable for a catalogue file.N/ADesirable
59System must allow users to upload updates on an already uploaded catalogue file.N/AMandatory
60System should allow only one active catalogue per Catalogue Workspace per EO.N/AImportant
61CWM users could be allowed to provide information regarding the Terms of Reference for the catalogues to be uploaded under the Catalogue Workspace.N/ADesirable
62CWM users should be able to associate more than one CA with a Catalogue Workspace.This will allow more than one CA to gain access to the items included in the catalogues uploaded in the Catalogue Workspace and perform purchases.Important
63CWM users should be able to associate more than one EO with a Catalogue Workspace.This will allow more than one EO to gain access to upload catalogue items in the Catalogue Workspace.Important
64EO users must be able to download the predefined catalogue template(s), available under the Catalogue Workspace, in order to prepare eCatalogues.A catalogue module may support several different templates based on which eCatalogues can be created.Mandatory
65System must allow access to a Catalogue Workspace only to users of organizations (CAs or EOs) that are associated with the specific workspace.N/AMandatory
66System should revoke access of users to a Catalogue Workspace that has reached its end date.N/AImportant

Catalogue browsing

No. Requirement Additional description Priority
67Users associated with a Catalogue Workspace must be able to view a Catalogue.EO users should only be able to view the Catalogue of their organization.Mandatory
68Users associated with a Catalogue Workspace must be able to view the list of Items included in a catalogue.EO users should only be able to view items contained in the Catalogue of their organization.Mandatory
69Users associated with a Catalogue Workspace must be able to view the details of Items included in a catalogue.EO users should only be able to view the details of items contained in the Catalogue of their organization.Mandatory
70CA Users could be allowed to compare Items.N/ADesirable
71CA Users could be allowed to create wishlists with Catalogue Items.N/ADesirable
72Users must be allowed to view the versioning history of a Catalogue.EO users should only be able to view the history of a Catalogue of their organization.Mandatory
73Users should be allowed to download the latest or historical versions of uploaded Catalogues.EO users should only be able to download Catalogues of their organization.Important
74Users should be allowed to download any files associated to a Catalogue.EO users should only be able to download files of a Catalogue of their organization. Such files may comprise:- Error report (reporting errors contained in a catalogue)- Warning report (reporting potential errors contained in a catalogue)- Comparison report (between two subsequent Catalogue versions).Important
75System should allow Requisitioner users to Punch-Out to an external site and return back with a filled in cart.As per the cXML punchout mechanism specifications.Important

Cart management

No. Requirement Additional description Priority
76Requisitioner users must be able to add Catalogue Items in Cart after accessing the details of an Item.N/AMandatory
77Requisitioner users should be able to add Catalogue Items in Cart after item search.N/AImportant
78Requisitioner users could be able to add Catalogue Items in Cart after browsing their wishlists.N/ADesirable
79System must calculate the total value of a Cart and display it to the Requisitioner.N/AMandatory
80Requisitioner users must be able to add an unlimited number of Items in the Cart.N/AMandatory
81System must inform users in case an already Cart-placed item (before creating a requisition) is no longer offered by the EO.N/AMandatory
82System must inform users in case the details of an already Cart-placed item (before creating a requisition) are changed.N/AMandatory
83Requisitioner users should be able to modify the Item quantities in the Cart and the Cart must be able to re-calculate the total value of the Items.N/AImportant
84Requisitioner users must be able to remove Items from Cart.N/AMandatory
85Requisitioner users must be able to "clear" all Cart Items.N/AMandatory
86Requisitioner users must be able to create a Requisition from a Cart.N/AMandatory

Catalogue Management

Catalogue preparation

No. Requirement Additional description Priority
87The Catalogue should be made available in a user-friendly and human-readable format (e.g. spreadsheet).Alternative technical mechanisms can be used (e.g. UBL standard, JSON/XML formats, etc.). Such approaches should be sought for advanced implementations, once adequate system/organizational maturity has been achieved.Important
88The Catalogue template must allow users to define all necessary aspects of each Catalogue Item.Indicative information comprises: ID/Stock-Keeping-Unit (SKU), Name, UNSPSC/CPV, Brand, Description, Unit customer price, Quantity in unit, Measurement unit, Unit list price, Image(s), URL, Manufacturer name, Manufacturer part number, Manufacturer URL, Warranty information, Product information sheet, Additional description, Environmental information, Lead delivery time, etc.Mandatory
89Provisions could be made so that the Catalogue template enables the user to provide Item information regarding Price Breaks and Item Attributes.Price Break: e.g. from quantity 1 to 100 price is X, while from quantity 101 to 1000 price is YItem Attributes: e.g. color choice for item "office chair".Desirable
90Catalogue template should indicate the minimum information required in order to be accepted as a valid Catalogue by the System, upon its submission.N/AImportant

Catalogue submission

No. Requirement Additional description Priority
91System must validate the structure of an uploaded Catalogue to ensure that it abides to the prescribed specifications.Structure validation concerns the technical format of the catalogue (e.g. file type).Mandatory
92In case an uploaded Catalogue does not abide to the prescribed specifications, system must inform the user accordingly.N/AMandatory
93The content of an uploaded Catalogue must be validated by the system to ensure that all 0 information is filled in.Content validation concerns the correctness of the catalogue in business terms (e.g. valid UNSPSC/CPV codes, valid prices, etc.).Mandatory
94In case 0 information is missing from a Catalogue, the system must inform the user accordingly, detailing the identified errors.The system could inform the user of any identified errors in the form of an Error Report.Mandatory
95System could notify via email the users assigned as Catalogue Approver users for the submission of a new or updated Catalogue.N/ADesirable
96System should enable the user to also upload additional Catalogue material (e.g. Item images, or manuals) along with the submission of the Catalogue.N/AImportant
97System could perform a quantitative/qualitative validation of an uploaded Catalogue and present to user potential business-context errors.Quantitative/qualitative validation could for instance inform users if a Catalogue Item UNSPSC/CPV code is not within the UNSPSC/CPV group code of the Catalogue workspace, or when the price of an Item is substantially higher/lower to the average of all Items. The system could inform the user of any potential errors in the form of a Warning Report.Desirable
98System must record the time of the Catalogue submission and the user that performed it.N/AMandatory
99Instead of uploading a Catalogue, the system should allow users to define the details of a Punchout connection.A Punchout Catalogue is a method through which CA Users can search items and create carts in an external website which can then be transferred back to the CA's own system for further processing (approval/ordering). A Punchout Catalogue follows standardized cXML specifications.Important

Catalogue approval

No. Requirement Additional description Priority
100A submitted Catalogue must be possible to be reviewed by CA users, before its Items are made available for purchasing.N/AMandatory
101To simplify the approval process, the system should mark the items that are New, Modified or Removed in a new Catalogue version versus its previous (active) version, to accommodate the review by the Catalogue Approver users.A Comparison Report could be created to report, per Catalogue ID / SKU, whether an item is changed, new, to be removed, or unchanged.Important
102A submitted Catalogue must be possible to be either approved or rejected by Catalogue Approver users.N/AMandatory
103The Items of an approved Catalogue must be made available to Requisitioner users for purchasing.N/AMandatory
104The Items of a rejected Catalogue must remain unavailable for purchasing.N/AMandatory

Catalogue versioning

No. Requirement Additional description Priority
105System must allow EO users (or CWM users on behalf of EOs) to upload a new version of a Catalogue with an intention to update an existing one.N/AMandatory
106A history of the Catalogue's revisions should be made available to users.N/AImportant
107Catalogue version should be automatically assigned by the System for newer versions of a Catalogue.N/AImportant

Catalogue activation

No. Requirement Additional description Priority
108Users should be able to define the Start /End date of a Catalogue during its submission.N/AImportant
109Catalogues with a start date in the future should be made active for purchasing once the define start date is reached.N/AImportant
110Catalogues should be deactivated once their end date is reached.N/AImportant

e-Purchasing

Requisition

No. Requirement Additional description Priority
111POM users must be able to approve or reject a Requisition.N/AMandatory
112POM users should be able to export a requisition in pre-specified format (e.g. CSV, spreadsheet).N/AImportant
113POM users should be able to add comments to a Requisition.N/AImportant
114Requisitioner users could be able to add non-Catalogue items to a Requisition.N/ADesirable

Quotation

No. Requirement Additional description Priority
115Requisitioner users could be able to create a Quotation request to be sent to the respective EO for obtaining a total value for purchase.A Quotation request forms a cart of items with negotiable item prices.Desirable
116EO users could be able to revise a Quotation request, and either reject it or submit a response.N/ADesirable
117Requisitioner users could be able to approve, reject or re-negotiate a Quotation response.N/ADesirable
118System could prepare a new Requisition upon approval of a Quotation.N/ADesirable
119System could support the exchange of Quotation data (i.e. requests and responses) between the e-Procurement and EO systems, through standardized means.The Request for Quotation and Quotation documents should abide to industry standards (e.g. UBL, cXML). The communication between the System and any other external systems shall be standardized thus enabling easy integration. The technologies to be used must ensure that no licensing costs shall burden any party.Desirable

Purchase Order

No. Requirement Additional description Priority
120System must be able to create a Purchase Order (PO) upon approval of a Requisition.N/AMandatory
121POM users must be able to approve or reject a PO.N/AMandatory
122POM users should be able to export a PO in pre-specified format (e.g. CSV, spreadsheet).N/AImportant
123System should be able to split a Requisition to one or more POs.A requisition may need to be split into several POs for several reasons, e.g. requisition contains items offered by different EOs, requisition contains items of different Catalogues/Contracts abiding to different purchasing terms, etc.Important

Invoice

No. Requirement Additional description Priority
124System must allow the creation of Invoices, linking them to POs.An "invoice" within an e-Procurement system may involve either the automated creation of invoice details through the integration with EO systems (abiding to a common standard) or a register of an already issued, paper-based invoice. In case full automation is sought, abidance to standards such as UBL should be scoped for implementation.Mandatory
125Each invoice must specify exact Item IDs / SKUs invoiced, along with pricing information.N/AMandatory
126POM users should be able to export an Invoice in pre-specified format (e.g. CSV, spreadsheet).N/AImportant

Payment

No. Requirement Additional description Priority
127System must allow the payment of an Invoice.A "payment" within an e-Procurement system may involve either the dispatch of a remittance advice for the electronic transfer of funds through integration with a Banking system or a register of an already performed payment through means outside the scope of the System. In case full automation is sought, abidance to standards such as UBL should be scoped for implementation.Mandatory
128POM users should be able to export a Payment in pre-specified format (e.g. CSV, spreadsheet).N/AImportant

Goods receipt note

No. Requirement Additional description Priority
129Requisitioner users must be able to create a Goods Receipt Note (GRN) and link it with an Invoice.GRN is the document in which the Requisitioner user reconciles the delivered goods against the invoiced ones and potentially registers damaged, missing or incorrect items.Mandatory
130Requisitioner users must be able to manage a GRN.N/AMandatory
131Requisitioner users must be able to finalize a GRN once all invoiced items are delivered and accepted.N/AMandatory

Non-functional

Availability

No. Requirement Additional description Priority
132System architecture must be designed in a manner that will ensure no single point-of-failure.- Redundancy of critical Hardware equipment should be ensured- Software design must ensure the appropriate use of the high-availability Hardware equipment (e.g. Clusterability of servers, fail-over support).Mandatory
133System architecture could make provisions for inter-connectivity with a Disaster Recovery site.N/ADesirable
134There must be an agreed minimum availability threshold, defined in the Service Level Agreement (SLA) between the service provider and public body responsible for the e-Procurement service delivery. This threshold must be met.Availability is calculated using the following formula:(Total Time of given period - Unplanned downtime) / Total Time of operation for given periodMandatory

Security

No. Requirement Additional description Priority
135All sensitive data stored in the various components of the system must be encrypted before stored.N/AMandatory
136System must support appropriate security controls, including user roles with pre-defined access rights which control the data and functionality each user has access to.N/AMandatory
137For all sensitive communications with clients, communication protocols with encryption (e.g. HTTPS) must be used.N/AMandatory
138System must provide anti-virus protection.N/AMandatory
139System must be protected against known security threats (e.g. SQL injection, DoS, DDoS, buffer overflows, XSS, etc.).N/AMandatory
140System must be able to guarantee that its data is not disclosed to unauthorized persons or processes.N/AMandatory

Auditability

No. Requirement Additional description Priority
141For critical system events (e.g. tender bid submission, auction bid submission, etc.), System must support methods by which the sender of data can be provided with proof of delivery.N/AMandatory
142System must be able to audit all 2 system and user actions.System should ensure that all actions performed on received/stored data are recorded, keeping track of actors, date/time, input/output data and any other information necessary to allow specialized personnel to monitor and fully reconstruct a transaction.Mandatory

Extendability

No. Requirement Additional description Priority
143System must be built in a modular approach that will allow the addition of new functional modules without impacting the overall system functionality.N/AMandatory
144System must be based in an architecture that will allow the addition of extra HW resources to enhance the systems capabilities (e.g. performance, storage, bandwidth, etc.).N/AMandatory

Portability

No. Requirement Additional description Priority
145System must be designed in a manner that will not be coupled to any hardware specific technologies.N/AMandatory
146System must be designed in a manner that will not be coupled to any proprietary software for its deployment and operation.N/AMandatory
147System must be possible to be deployed on different HW and SW infrastructures.N/AMandatory

Reliability

No. Requirement Additional description Priority
148System must ensure the integrity of the transmitted data.N/AMandatory

Usability

No. Requirement Additional description Priority
149System should abide to the latest standards for Graphical User Interface design.N/AImportant
150System must be accessible over the web through the use of any available web-browser.System should at a minimum support all versions of the most widely-used web-browsers that the relevant vendors continue to support.Mandatory
151System could be able to operate and its content to be possible to be made available in different languages.N/ADesirable

Performance

No. Requirement Additional description Priority
152System must be able to cope with the expected load and traffic.N/AMandatory
153System's architecture should be able to cover an annual increase of at least an agreed percentage on the load and traffic without degradation of its performance.N/AImportant
154System response times (in a LAN environment) should not exceed the agreed number of seconds for responding to user requests. Agreed number of seconds should be split per type of functionality such as simple queries, complex queries, reports, etc.N/AImportant

Interoperability

No. Requirement Additional description Priority
155System must follow state-of-the-art interoperability standards so that its integration or communication with external systems can be achieved.Such integration points include but are not limited to:- Quotation requests / responses- Purchase orders- Invoices- Payments- Authentication and Single Sign On services- National Supplier Registry system etc.Mandatory
